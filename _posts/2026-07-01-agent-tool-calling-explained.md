学 Agent 时最容易被一堆名词绕晕：Function Calling、MCP、Skills、Subagent……

我水平有限，不搞论文式解释，只写 **我实际用下来理解到的部分**。

## 核心就一件事：模型本身不能「动手」

大模型只会生成文字。  
要让它 **读你的项目、改文件、执行命令**，必须接 **工具（Tools）**。

流程可以简化成：

```
你说：帮我在博客加一篇文章
    ↓
Agent 规划：先看 _posts 格式 → 再新建 md → 再检查 front matter
    ↓
调用工具：Read / Write / Grep / Shell …
    ↓
把结果拼进上下文，继续下一步
    ↓
输出给你：改好了，路径是 xxx
```

所以 Agent 强不强，**一半看模型，一半看工具和环境给不给力**。

## 我接触过的几类工具

| 类型     | 干什么              | 我遇到的例子       |
| -------- | ------------------- | ------------------ |
| 读文件   | 看代码、配置        | Read               |
| 写文件   | 创建/修改           | Write / StrReplace |
| 搜索     | 在仓库里找符号      | Grep / Glob        |
| 终端     | 跑 git、npm         | Shell              |
| 外部服务 | 接 GitHub、数据库等 | MCP                |

**MCP** 我理解为：给 Agent 接「插件接口」的标准方式。  
比如接 Supabase、Figma——不用你手动复制粘贴，Agent 能直接调。

## 为什么有时 Agent 会「瞎编」

常见原因之一：**该用工具时没用上，或工具失败了它还在编。**

比如：

- 没读文件就假设目录结构
- 命令失败了但总结里写「已成功」
- 搜不到代码却编造函数名

所以我现在的习惯：**关键结论让它「引用的路径/命令输出」可见**，不能只有一句「搞定了」。

## 和 Chat 的差别（再强调一次）

Chat：知识主要在训练数据里，**碰不到你本机仓库**。  
Agent：通过工具 **绑定你的真实环境**。

这也是为什么 Agent 更适合 **工程任务**，Chat 更适合 **概念问答**。
