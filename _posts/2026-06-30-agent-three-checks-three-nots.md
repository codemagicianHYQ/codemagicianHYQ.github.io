从五月底到六月，我越来越依赖 Agent 写代码、改博客、折腾 side project。

效率确实上去了，但也出现过：**一次对话改乱五个文件、终端命令跑错目录** 的情况。

现在我给自己定了一套很土但管用的规矩，叫 **「三看三不」**。

## 三看（Agent 干完必做）

### 1. 看 diff

不只看总结，**逐文件看改了什么**。  
尤其警惕：意外改动的 `package.json`、`.env`、全局配置。

### 2. 看终端输出

Agent 说「已修复」不算数，**报错还在不在**才算数。  
能跑 `npm run dev` / `git status` 就自己扫一眼。

### 3. 看范围

问一句：**它有没有动到任务范围外的东西？**  
写博客却改了 `_config.yml`，写前端却碰了后端——都要警惕。

## 三不（交给 Agent 之前）

### 1. 不上来就「全权委托」

大重构、删库、force push 这类，**我自己动手或分步确认**。

### 2. 不给模糊目标

❌ 「把这个项目弄好」  
✅ 「在 `admin/config.yml` 加封面图字段，别改别的」

### 3. 不跳过 Git

Agent 改之前：**先 commit 或 stash**。  
改坏了能回滚，心理负担小很多。

## 一个真实场景

我在博客后台加 CMS 配置时，Agent 很快生成了 `admin/config.yml`。  
但我自己还是检查了：仓库名、分支名、media 路径——有一处 remote 写错的话，后台登录会莫名其妙失败。

**Agent 提速，人来做最后一道质检。**

## 写在最后

Agent 的价值，我觉得不是「少写代码」，而是 **「少在低价值重复上耗神」**。

但质检不能省。  
对我这种还在实习、还在学的人来说，**会审 Agent 的输出**，可能比会写 prompt 更重要。
