六月用 Agent 多了以后，我慢慢分出两类任务：

- 交给它，**省时间且风险可控**
- 必须自己来，**错了代价太大或它做不好**

这不是标准答案，只是 **实习 + 自学阶段** 的我，真实在用的划分。

## 我会交给 Agent 的

### 1. 样板和重复劳动

- 新建博客文章 front matter
- 批量改 import 路径
- 根据报错日志定位可能文件（再自己确认）

### 2. 读陌生代码

clone 一个新项目（比如 Electron 桌面端），先让 Agent **梳理目录、入口、依赖关系**。  
比我从零啃 README 快，但跑起来还得自己 `npm i`。

### 3. 文档和 commit 初稿

博客正文我会先列大纲；  
commit message 可以让 Agent 根据 `git diff` 起草，**我改几个字再提交**。

## 我坚决自己来的

### 1. 业务判断

「这个功能要不要做」「接口字段含义是什么」——得问产品/后端/自己查需求，**Agent 猜不准**。

### 2. 安全和密钥

`.env`、Token、SSH key、数据库密码。  
Agent 可能误提交或写进日志，**这类文件我手动处理**。

### 3. 线上和 Git 危险操作

`push --force`、删分支、改生产配置。  
最多让 Agent **生成命令**，执行前我自己再看一遍。

### 4. 需要背锅的事

线上 bug、给客户/demo 的版本——**最终 quality bar 在我**，不能「Agent 说的我就上了」。

## 一个中间地带：Agent 起草，人拍板

比如博客改 Chirpy 主题、加 CMS 后台：

- Agent 可以生成配置和文件
- **是否 push、是否改 PWA、是否动 `_config.yml` 核心项**——我自己决定

这种 **「AI 执行 + 人决策」** 模式，目前最适合我。

## 七月的小目标

1. 把 Agent 工作流写进固定 checklist（开任务前 / 结束后各 2 分钟）
2. 练 **小步提交**：Agent 每完成一小步就 commit，方便回滚
3. 记录失败案例：哪类任务 Agent 特别容易翻车

Agent 是杠杆，不是替身。  
**杠杆用在对的地方才省力。**
